# Aircraft Performance Data Files — Design Spec

## Problem

Aircraft performance data is hardcoded in `aircraft.rs` as 6 static presets with limited fields. This makes it difficult to add new aircraft, prevents community contributions, and constrains the flight model to simplified calculations. More detailed performance data (per-phase speeds, fuel flow, service ceiling) is needed to improve flight plan accuracy.

## Goals

1. **Richer built-in database**: ~14 aircraft profiles with detailed per-phase performance data, embedded at compile time
2. **User-customizable**: Runtime loading of custom TOML profiles via CLI flag
3. **Import from external sources**: CLI subcommand to convert LNM `.lnmperf` files (extensible to other formats)
4. **AI-authorable**: Format documentation that enables an AI to produce valid profiles from any performance data source

## Non-Goals

- Altitude-band dependent performance profiles (future extension)
- Weight-dependent performance modeling (future extension)
- Fuzzing the TOML parser (add when accepting untrusted external input)
- Network download of performance data at build time (unlike airports, profiles are committed)

## TOML File Format

Each aircraft is a single `.toml` file in `data/aircraft/`, named by lowercase ICAO type designator (e.g., `b738.toml`).

```toml
[aircraft]
name = "Boeing 737-800"       # Human-readable name
icao_type = "B738"             # ICAO type designator (lookup key)

[performance]
cruise_speed_ktas = 460        # True airspeed in cruise (knots)
cruise_altitude_ft = 36000     # Optimal cruise altitude (feet)
service_ceiling_ft = 41000     # Maximum certified altitude (feet)
min_runway_length_ft = 6000    # Minimum runway length (feet)

[performance.climb]
speed_ktas = 310               # Climb TAS (knots)
rate_fpm = 2500                # Climb rate (feet per minute)

[performance.descent]
speed_ktas = 280               # Descent TAS (knots)
rate_fpm = 1800                # Descent rate (feet per minute)

[fuel]
capacity_kg = 20894.0          # Usable fuel capacity (kilograms)
fuel_type = "jet"              # "jet" (Jet-A, 0.804 kg/L) or "avgas" (0.721 kg/L)

[fuel.flow]
climb_kg_per_hour = 3402.0     # Fuel flow during climb
cruise_kg_per_hour = 2359.0    # Fuel flow during cruise
descent_kg_per_hour = 1270.0   # Fuel flow during descent
```

### Key Decisions

- **ICAO type as identifier**: Matches aviation standards and LNM's `AircraftType` field. Used for CLI `--aircraft` lookup.
- **Fuel normalized to kg**: International aviation standard (ICAO). Import tools convert from lbs, gal, or litres using fuel density.
- **Absolute climb/descent speeds**: Replaces the current `climb_speed_factor`/`descent_speed_factor` approach. More accurate and directly sourced from performance data.
- **Range is derived**: Computed as `(capacity_kg / cruise_kg_per_hour) * cruise_speed_ktas` instead of a hardcoded field. More accurate since it's based on actual fuel data.
- **Service ceiling**: Enables future altitude selection logic and validates cruise altitude.

### Future Extension Points

The format can be extended without breaking existing files:

- **Altitude bands**: Optional `[[performance.climb.bands]]` array alongside flat fields
- **Weight classes**: Optional `[weights]` section (OEW, MTOW, MLW)
- **Speed schedules**: Optional `[performance.speed_schedule]` for 250kt-below-FL100 and IAS/Mach crossover

## Data Model Changes

The `Aircraft` struct evolves from the current 8 fields:

### Current

```rust
pub struct Aircraft {
    pub name: &'static str,
    pub cruise_speed_kts: u16,
    pub cruise_altitude_ft: u32,
    pub climb_rate_fpm: u16,
    pub descent_rate_fpm: u16,
    pub climb_speed_factor: f32,
    pub descent_speed_factor: f32,
    pub range_nm: u32,
    pub min_runway_length_ft: u32,
}
```

### New

Two struct variants are used: `&'static str` for compile-time embedded data (generated by `build.rs`), and a deserialization struct for TOML parsing that converts into the core type.

```rust
/// Core struct used throughout the codebase. Built-in profiles use 'static refs.
pub struct Aircraft {
    pub name: &'static str,
    pub icao_type: &'static str,
    pub cruise_speed_ktas: u16,
    pub cruise_altitude_ft: u32,
    pub service_ceiling_ft: u32,
    pub min_runway_length_ft: u32,
    pub climb_speed_ktas: u16,
    pub climb_rate_fpm: u16,
    pub descent_speed_ktas: u16,
    pub descent_rate_fpm: u16,
    pub fuel_capacity_kg: f64,
    pub fuel_type: FuelType,
    pub fuel_flow_climb_kg_per_hour: f64,
    pub fuel_flow_cruise_kg_per_hour: f64,
    pub fuel_flow_descent_kg_per_hour: f64,
}

pub enum FuelType {
    Jet,    // Jet-A, density 0.804 kg/L
    Avgas,  // 100LL, density 0.721 kg/L
}

/// Owned variant for runtime-loaded custom profiles.
/// Converts into Aircraft via leaking strings into 'static refs,
/// which is acceptable since custom profiles are loaded once at startup.
pub struct AircraftOwned {
    pub name: String,
    pub icao_type: String,
    // ... same numeric fields as Aircraft
}

impl AircraftOwned {
    pub fn into_static(self) -> Aircraft { /* leak strings */ }
}
```

The TOML deserialization uses nested serde structs that mirror the file structure:

```rust
#[derive(Deserialize)]
struct AircraftToml {
    aircraft: AircraftMeta,
    performance: PerformanceToml,
    fuel: FuelToml,
}

#[derive(Deserialize)]
struct AircraftMeta { name: String, icao_type: String }

#[derive(Deserialize)]
struct PerformanceToml {
    cruise_speed_ktas: u16,
    cruise_altitude_ft: u32,
    service_ceiling_ft: u32,
    min_runway_length_ft: u32,
    climb: PhaseToml,
    descent: PhaseToml,
}

#[derive(Deserialize)]
struct PhaseToml { speed_ktas: u16, rate_fpm: u16 }

#[derive(Deserialize)]
struct FuelToml {
    capacity_kg: f64,
    fuel_type: String,
    flow: FuelFlowToml,
}

#[derive(Deserialize)]
struct FuelFlowToml {
    climb_kg_per_hour: f64,
    cruise_kg_per_hour: f64,
    descent_kg_per_hour: f64,
}
```

### Migration

- `climb_speed_factor` / `descent_speed_factor` removed — replaced by absolute `climb_speed_ktas` / `descent_speed_ktas`
- `range_nm` removed — derived from fuel data via `Aircraft::range_nm()` method, which applies a 5% reserve factor: `(capacity_kg * 0.95 / cruise_kg_per_hour) * cruise_speed_ktas`. Values validated against current presets to prevent regression.
- `name` split into `name` (human-readable) and `icao_type` (lookup key). `aircraft_by_name()` renamed to `aircraft_by_icao_type()`.
- Flight model in `flight_plan.rs` updated to use new field names (direct substitution, no logic change needed for the current calculation model)
- The existing inline custom aircraft CLI flags (`--cruise-speed`, `--cruise-alt`, etc.) are removed. Users should use `--profile path/to/custom.toml` instead. This avoids an explosion of required flags for fuel fields and keeps the CLI clean.

## Build Pipeline

### Built-in Profiles (Compile-Time)

1. TOML files committed in `data/aircraft/*.toml`
2. `build.rs` extended to:
   - Glob all `data/aircraft/*.toml` files
   - Parse each with `toml` crate + serde
   - Validate schema: required fields present, sane ranges (speeds > 0, ceiling >= cruise altitude, fuel capacity > 0), and no duplicate `icao_type` across files
   - The `icao_type` field inside the TOML is authoritative; the filename is for human convenience only but must match (lowercase icao_type + `.toml`) — build fails on mismatch
   - Generate `aircraft_db.rs` with a static array of `Aircraft` structs
3. Included via `include!()` in `aircraft.rs` — same pattern as `airport_db.rs`
4. Rebuild triggered when any file in `data/aircraft/` changes

### Custom Profiles (Runtime)

- New CLI flag: `--profile path/to/custom.toml`
- Parsed at runtime using the same TOML deserialization into `AircraftOwned`, then converted to `Aircraft` via `into_static()`
- When `--profile` is provided, that aircraft is always used — `--aircraft` is ignored (or forbidden). This avoids silent ICAO type mismatches.
- Uses the `toml` crate (added as a non-build-only dependency)

## CLI Changes

```
random-flight generate --aircraft B738 --time 4h
                       [--profile path/to/custom.toml]

random-flight aircraft list                      # List built-in presets

random-flight aircraft import                   # Import from external format
    --format <lnmperf|...>                      # Source format (required)
    <input>                                     # Input file path
    [--output <dir>]                            # Output dir (default: ./)
```

## LNM Import Subcommand

`random-flight aircraft import --format lnmperf <file>`

### Parsing

Supports both LNM format versions:
- **XML format** (v2.6+): Parse `<LittleNavmap><AircraftPerf>` structure
- **INI format** (v2.4.x): Parse `[Options]` and `[Perf]` sections

### Conversion

- **Fuel normalization**: Convert lbs or gal to kg using fuel density (`FuelAsVolume` + `JetFuel` flags determine source unit)
- **Speed mapping**: LNM provides per-phase TAS directly — maps 1:1
- **Field mapping**:

| LNM Field | TOML Field |
|---|---|
| `AircraftType` | `aircraft.icao_type` |
| `Name` | `aircraft.name` |
| `Climb/SpeedKtsTAS` | `performance.climb.speed_ktas` |
| `Climb/VertSpeedFtPerMin` | `performance.climb.rate_fpm` |
| `Cruise/SpeedKtsTAS` | `performance.cruise_speed_ktas` |
| `Descent/SpeedKtsTAS` | `performance.descent.speed_ktas` |
| `Descent/VertSpeedFtPerMin` | `performance.descent.rate_fpm` |
| `MinRunwayLengthFt` | `performance.min_runway_length_ft` |
| `UsableFuelLbsGal` | `fuel.capacity_kg` (after conversion) |
| `Climb/FuelFlowLbsGalPerHour` | `fuel.flow.climb_kg_per_hour` (after conversion) |
| `Cruise/FuelFlowLbsGalPerHour` | `fuel.flow.cruise_kg_per_hour` (after conversion) |
| `Descent/FuelFlowLbsGalPerHour` | `fuel.flow.descent_kg_per_hour` (after conversion) |

- **Missing fields**: `service_ceiling_ft` and `cruise_altitude_ft` are not in LNM format. The import tool outputs estimated defaults with a comment noting they should be verified: `cruise_altitude_ft` defaults to 36000 for jets / 10000 for piston, and `service_ceiling_ft` defaults to `cruise_altitude_ft + 5000`. The output file is written to the `--output` directory (default: current directory, NOT `data/aircraft/`) so it doesn't break the build until the user reviews and moves it.

### Output

Writes a `.toml` file in our canonical format. The user reviews/edits before committing.

## FORMAT.md

A `data/aircraft/FORMAT.md` document that serves as both human and AI reference:

- Complete schema with all fields, types, units, and valid ranges
- Field-by-field descriptions with guidance on sourcing values
- Annotated example profile
- Conversion reference (lbs/kg, gal/L, fuel densities)
- Tips for common aircraft categories (GA vs jet performance ranges)

This enables an AI to produce valid profiles when given the document plus a source of performance data.

## Testing

- **Schema validation**: Parse each built-in TOML file, verify required fields present and within sane ranges (speeds > 0, ceiling >= cruise altitude, fuel capacity > 0, etc.)
- **Round-trip**: Ensure build.rs-generated structs match runtime TOML parsing for the same file
- **LNM import**: Parse sample `.lnmperf` files (both XML and INI), verify unit conversion (lbs/gal to kg) and output validity
- **Range computation**: Verify derived range matches expected values for known aircraft
- **Custom profile**: Load a `--profile` override at runtime, verify it takes precedence over built-in
- **Regression**: Existing integration tests pass — flight model uses same inputs, sourced differently

## Dependencies

New crate dependencies:
- `toml` — TOML parsing (build-time and runtime)
- `quick-xml` or `xml-rs` — XML parsing for LNM import

Both must be added explicitly to `Cargo.toml` (build-dependencies for `toml` in build.rs, regular dependency for runtime loading; `quick-xml` as a regular dependency for the import subcommand).
